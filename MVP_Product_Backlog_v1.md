## 3. User Stories (для P0 Features)

Формат: **Как `<роль>` я хочу `<действие>`, чтобы `<ценность>`** + критерии приёмки.

**US-1 (F1.1).** Как гражданин я хочу войти по номеру телефона и SMS-коду, чтобы начать
пользоваться сервисом без регистрации на госпорталах.
*Критерии:* код приходит за ≤30с; неверный код → `invalid_otp`; после 3 попыток — лимит;
успех выдаёт JWT.

**US-2 (F1.3).** Как гражданин я хочу заполнить и отредактировать профиль, чтобы система
знала мои данные для подбора.
*Критерии:* обязательны ФИО и регион; изменения сохраняются; телефон меняется через верификацию.

**US-3 (F2.1).** Как гражданин я хочу создать кейс по жизненной ситуации, чтобы начать
сопровождение.
*Критерии:* выбор `life_event`; повторный активный кейс по той же ситуации → `409`.

**US-4 (F2.2/F2.3).** Как гражданин я хочу пройти короткую анкету с уточнениями под мою
ситуацию, чтобы получить точный подбор без лишних вопросов.
*Критерии:* базовая ≤15 вопросов; динамика зависит от `life_event`; автосохранение; прогресс.

**US-5 (F3.1).** Как гражданин я хочу увидеть список положенных мер, чтобы понять, что мне
доступно.
*Критерии:* результат ≤3с; группировка eligible/conditional/need_info; hidden скрыт;
пишется RuleEvaluationLog.

**US-6 (F3.2).** Как гражданин я хочу понять, почему мера подходит и чего не хватает, чтобы
доверять рекомендации.
*Критерии:* показаны matched, missing, next_step, required_documents; доступно в 1 клик.

**US-7 (F4.1/F4.2).** Как гражданин я хочу загрузить документы по чек-листу меры, чтобы
подготовиться к подаче.
*Критерии:* поддержка PDF/JPG/PNG; лимит размера; отметка наличия по `doc_type`.

**US-8 (F5.1/F5.2).** Как гражданин я хочу подать заявку на меру и видеть её статус, чтобы
отслеживать ход.
*Критерии:* фиксируется `measure_version`; без обязательных документов → `422`; статусы из
справочника; переход пишется в историю.

**US-9 (F6.1).** Как гражданин я хочу получать уведомления об изменениях, чтобы не пропустить
важное.
*Критерии:* внутрисистемный список; отметка прочтения.

**US-10 (F7.1/F7.2/F7.3).** Как администратор я хочу загрузить и версионировать меры и
справочники, чтобы наполнять платформу без разработчиков.
*Критерии:* импорт таблицы; новая версия не ломает старые заявки (`is_current`); RBAC — только admin.

**US-11 (F9.2/F9.3).** Как система я должна защищать персональные данные и разграничивать
доступ, чтобы соответствовать 152-ФЗ.
*Критерии:* шифрование в покое/транзите; координатор видит только своих; audit log; 2FA для coord+.

## 4. Дорожная карта по спринтам

Предпосылка (уточнено основателем): команда **2 человека** (Артур + ещё один), спринт 2 недели.
Из-за малой команды объём каждого спринта сокращён, а сроки растянуты — акцент строго на
P0-фичах и на **демонстрируемом сквозном сценарии** «вход → анкета → подбор → объяснение →
заявка» (нужен рабочий прототип для показа заказчику). P1/P2 откладываются до после демо.

**Редакция v1.2 (уточнено PM под команду из 2 человек, ~13 недель до демо-MVP; ИИ как «третий
член команды»):** акцент на ОДНОМ безупречном сквозном сценарии, а не на охвате всех функций.

| Спринт | Длит. | Цель | Состав | Демонстрируемый результат |
|--------|-------|------|--------|---------------------------|
| **Sprint 0** | ~1–2 нед | Фундамент | Авторизация-скелет, каркас Backend (модульный монолит), каркас Frontend, схема БД, CI/CD | Пустое приложение деплоится, схема поднята |
| **Sprint 1** | 3 нед | Вход и анкета | SMS, профиль, создание кейса, выбор life_event, базовая + динамическая анкета | Гражданин входит и проходит анкету |
| **Sprint 2** | 3 нед | Ядро ценности | Rule Engine, импорт мер Краснодарского края, подбор, Explainability | Гражданин видит меры и объяснение «почему» |
| **Sprint 3** | 3 нед | Документы, заявки, координатор | Загрузка документов, создание заявки, статусы, кабинет координатора | Гражданин подаёт заявку, координатор её видит |
| **Sprint 4** | 2 нед | Полировка и пилот | UX-полировка, обработка ошибок, сборка ДЕМО, запуск пилота | Готовое демо + пилот в КК на реальных льготах |

Итого ~**13 недель** до демонстрационного MVP — реалистично для двоих при активном использовании ИИ.

*(Предыдущая 6-спринтовая раскладка v1.0 сохранена в истории; v1.2 — рабочая.)*

## 5. Definition of Done

Задача считается выполненной, если:
- код реализован по спецификациям и покрыт автотестами (unit + ключевые интеграционные);
- соответствует API-контрактам (`MVP_API_Contracts_v1.md`) — проверено контрактными тестами;
- обновлена документация (при изменении контракта/модели);
- пройден code review минимум одним инженером;
- выполнены проверки безопасности (RBAC, доступ к чужим данным, валидация ввода);
- сценарии User Flow из `MVP_User_Flows_v1.md` работают без регрессий (e2e по ключевому пути).

## 6. Риски и меры снижения

| Риск | Влияние | Мера снижения |
|------|---------|---------------|
| Интеграции с госсистемами (Госуслуги/СФР/МО) | Год 1 работает без них | MVP на SMS + ручной ввод/импорт; интеграции — год 2–3, абстракция провайдера уже заложена |
| Качество данных о мерах поддержки | Неверный подбор подрывает доверие | Ручная валидация при импорте; версионирование; журнал оценки для разбора; пилот на выверенном наборе Краснодарского края |
| Изменения законодательства | Меры устаревают | Версионирование мер (`measure_version`), редактирование справочников без кода, `valid_from/to` |
| Производительность Rule Engine | Медленный подбор (>3с) | Дешёвая фильтрация по индексам до дорогой проверки; GIN по `eligibility`; кэш результатов по версии профиля |
| Безопасность персональных данных (152-ФЗ) | Юридические и репутационные риски | Шифрование AES-256/TLS 1.3, RBAC, audit log, данные в РФ, 2FA для координаторов, проверки в DoD |
| Разрастание scope MVP | Срыв сроков | Жёсткий P0/P1/P2; P1/P2 не входят в спринты 0–3; пилот на минимальном наборе |

## 8. Решения основателя (см. _ОБМЕН/DECISIONS.md)

1. **Команда:** 2 человека (Артур + ещё один). Scope спринтов сокращён, акцент на P0 и
   демонстрируемом сквозном сценарии.
2. **Пилотный регион:** Краснодарский край — подтверждён.
3. **Метрики успеха пилота:** число поданных заявок; удовлетворённость граждан и координаторов;
   передача успешной истории между регионами (референс среди министерств/координаторов).
4. **Целевая категория:** участники СВО и семьи; далее расширение на инвалидов и иные соц.
   уязвимые группы.
5. **Цель этапа:** донести ценность заказчику + рабочий демонстрируемый прототип.
6. **Следующий шаг:** техническая архитектура сервисов (декомпозиция Backend/Frontend/БД).
