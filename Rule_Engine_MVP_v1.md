## 0. Назначение

Документ описывает архитектуру принятия решений, по которой «КУРАТОР» автоматически
подбирает гражданину (или организации) подходящие меры поддержки, ранжирует их и объясняет
каждый результат. Это главное интеллектуальное ядро MVP — механизм создания ценности.

**Принцип:** правила — это данные, а не код. Rule Engine интерпретирует декларативные
критерии (`eligibility`) из карточки меры и профиль заявителя. Добавление/изменение меры не
требует деплоя. Архитектура спроектирована так, чтобы в будущем заменить JSON-интерпретатор
полноценным Rule Engine (Drools/собственный) без изменения модели данных.

## 1. Общая архитектура подбора

Подбор — детерминированный конвейер (pipeline) из 7 стадий:

```
        ┌─────────────┐
        │ Пользователь│
        └──────┬──────┘
               ▼
        ┌─────────────┐   Профиль + ответы анкеты (Project.life_event,
        │   Анкета     │   категории, регион, доход, дети, ОКВЭД и т.д.)
        └──────┬──────┘
               ▼
        ┌──────────────────┐  Приведение к каноническому Evaluation Context:
        │  Нормализация    │  типы, единицы, вычисляемые поля (возраст из birth_date)
        └──────┬───────────┘
               ▼
        ┌──────────────────┐  Грубый отбор по индексируемым полям:
        │   Фильтрация     │  регион, тип, is_current, дата действия. Дёшево, по БД.
        └──────┬───────────┘
               ▼
        ┌──────────────────┐  Полная проверка eligibility каждой прошедшей меры:
        │ Проверка критериев│  AND / OR / NOT / диапазоны / списки / булевы
        └──────┬───────────┘
               ▼
        ┌──────────────────┐  Расчёт Score (0–100) + категория результата
        │   Расчёт Score   │  (eligible / conditional / need_info / hidden)
        └──────┬───────────┘
               ▼
        ┌──────────────────┐  Сортировка по Score, затем по приоритету уровня
        │   Ранжирование   │  (federal < regional < municipal перекрытие) и сумме
        └──────┬───────────┘
               ▼
        ┌──────────────────┐  Список рекомендаций + Explainability по каждой
        │ Вывод рекомендаций│
        └──────────────────┘
```

**Свойства конвейера:**

- **Детерминированность** — одинаковый вход всегда даёт одинаковый выход (требование для
  государственного заказчика и аудита).
- **Дешёвая фильтрация до дорогой проверки** — стадия «Фильтрация» отсекает 95% мер по
  индексам БД (регион/тип/дата), «Проверка критериев» работает уже с десятками, не тысячами.
- **Идемпотентность и кэшируемость** — результат подбора привязан к версии профиля и версиям
  мер; при неизменном входе берётся из кэша.
- **Аудируемость** — каждый прогон логируется (вход, версии правил, результат) — совместимо с
  audit log из архитектуры.

## 2. Источники данных (Evaluation Context)

Нормализатор собирает единый плоский контекст оценки из сущностей модели данных v1.1:

| Источник | Поля | Роль в подборе |
|----------|------|----------------|
| `user` | `birth_date`→`age`, `region_id`, `status` | Базовый профиль гражданина |
| `user_category` → `citizen_category` | `code[]` (список кодов категорий) | Главный вход: СВО, инвалид, многодетный… |
| `organization` | `type`, `region_id` | Для мер B2G/организациям |
| `project` | `life_event`, `region_id`, `status` | Контекст жизненной ситуации |
| `region` | `code` | Территориальная применимость |
| `support_measure` | `eligibility`, `level`, `region_id`, `valid_from/to`, `is_current` | Проверяемый объект |
| Анкета (ad-hoc) | `income`, `children_count`, `disability`, `okved`, `project_stage`, `funding_amount`, `industry`… | Данные, которых нет в БД, вводятся в форме |

**Канонический Evaluation Context** (пример):

```json
{
  "subject_type": "citizen",
  "age": 36,
  "region_code": "23",
  "categories": ["svo_participant", "large_family"],
  "income_month": 45000,
  "children_count": 3,
  "disability_group": null,
  "life_event": "svo_return",
  "org_type": null,
  "okved": null,
  "project_stage": null,
  "funding_amount": null,
  "as_of_date": "2026-07-06"
}
```

Нормализация отвечает за: вычисление `age` из `birth_date`, приведение единиц (доход —
руб/мес), заполнение отсутствующих полей `null` (важно для стадии `need_info`), сбор списка
`categories` из `user_category`.

## 3. Структура `eligibility` — универсальный шаблон

Единый расширяемый JSON-формат, хранится в `support_measure.eligibility` (JSONB). Состоит из
логических блоков `all` (AND), `any` (OR), `none` (NOT) и листовых условий.

```json
{
  "version": 1,
  "subject_type": "citizen",
  "rules": {
    "all": [
      { "field": "categories", "op": "contains_any", "value": ["svo_participant"] },
      { "field": "region_code", "op": "in", "value": ["23"] },
      { "field": "age", "op": "between", "value": [18, 65] }
    ],
    "any": [
      { "field": "children_count", "op": "gte", "value": 3 },
      { "field": "disability_group", "op": "in", "value": ["I", "II"] }
    ],
    "none": [
      { "field": "categories", "op": "contains_any", "value": ["excluded_flag"] }
    ]
  },
  "soft": [
    { "field": "income_month", "op": "lte", "value": 50000, "weight": 20 }
  ],
  "required_fields": ["categories", "region_code", "age"]
}
```

Пояснения:
- `rules.all` — жёсткие обязательные условия (все должны выполниться).
- `rules.any` — достаточно одного из.
- `rules.none` — исключающие (ни одно не должно выполниться).
- `soft` — необязательные условия, влияющие только на Score (приоритет, а не допуск).
- `required_fields` — какие данные должны присутствовать в контексте; если их нет → `need_info`.

**Поддерживаемые типы полей:** число, строка, дата, булево, список (categories), диапазон.
Шаблон расширяем: новое поле подбора = добавить ключ в контекст (§2) и использовать в правиле.
Изменение кода не требуется.

## 4. Алгоритм проверки

Для каждой меры, прошедшей грубую фильтрацию, вычисляется результат каждого листового условия.

### 4.1 Операторы (`op`)

| Категория | Операторы | Применение |
|-----------|-----------|-----------|
| Равенство | `eq`, `neq` | регион, форма организации |
| Списки | `in`, `not_in`, `contains_any`, `contains_all` | категории, ОКВЭД |
| Диапазоны | `gt`, `gte`, `lt`, `lte`, `between` | возраст, доход, сумма |
| Булевы | `is_true`, `is_false` | флаги (инвалидность, наличие детей) |
| Существование | `exists`, `not_exists` | обязательность данных анкеты |

### 4.2 Логика агрегации

```
pass_hard = ALL(rules.all) AND ANY(rules.any, если блок задан) AND NONE(rules.none)
```

- Если какое-то `required_field` в контексте `null`/отсутствует → результат меры =
  **`need_info`** (не отбраковываем, а помечаем «нужны данные»).
- Если `pass_hard = false` из-за жёсткого условия → мера **`hidden`** (не показываем) — с
  сохранением причины во внутреннем логе (для аудита, не для гражданина).
- Если `pass_hard = true` → мера кандидат, идёт на расчёт Score.

### 4.3 Обработка пустых блоков

Отсутствующий блок `any`/`none` считается пройденным (нейтральным). Пустой `all` = мера
доступна всем (федеральная база) — допускается сознательно.

## 5. Система Score (0–100)

Score отражает степень соответствия и уверенность в рекомендации. Формула прозрачна и
объяснима (никакого «чёрного ящика» на этапе MVP).

```
base = 100                                  # прошёл все жёсткие критерии
- penalty_need_info (по 20 за каждое отсутствующее required_field)
+ soft_bonus (сумма weight выполненных soft-условий, кап +0, только для сортировки)
```

**Категории результата (по Score):**

| Score | Категория | Что показываем гражданину |
|-------|-----------|---------------------------|
| 100 | `eligible` — полностью соответствует | «Вам положено. Можно подавать заявку» |
| 80–99 | `conditional` — соответствует с уточнениями | «Скорее всего положено, уточните N пунктов» |
| 60–79 | `need_info` — требуется доп. информация | «Ответьте на N вопросов для проверки» |
| < 60 | `hidden` | не показывается |

**Обоснование подхода:** пороговая шкала с явными правилами (а не ML-скор) выбрана потому, что
на этапе MVP и при работе с госзаказчиком важнее **объяснимость и юридическая
воспроизводимость**, чем точность вероятностной модели. Каждый балл трассируется до
конкретного условия. `soft`-условия влияют только на порядок внутри группы `eligible`, не
понижая ниже допуска, — чтобы «более релевантные» меры были выше.

**Ранжирование внутри выдачи:** (1) по категории (eligible → conditional → need_info),
(2) по Score убыв., (3) по перекрытию уровней (муниципальная/региональная выше федеральной
при равенстве — как в архитектуре: приоритет более локальных правил), (4) по сумме `amount`.

## 6. Explainability (объяснимость)

Обязательное требование доверия. Для каждой рекомендации движок возвращает структуру
объяснения — не только «да/нет», а «почему».

```json
{
  "measure_id": "…",
  "measure_code": "KK-SVO-PAY-001",
  "score": 80,
  "category": "conditional",
  "matched": [
    { "field": "categories", "rule": "contains_any [svo_participant]", "value": ["svo_participant"], "ok": true },
    { "field": "region_code", "rule": "in [23]", "value": "23", "ok": true }
  ],
  "unmet": [],
  "missing": [
    { "field": "income_month", "reason": "нужно указать доход для проверки soft-критерия" }
  ],
  "next_step": "Укажите ежемесячный доход, чтобы подтвердить право и подать заявку",
  "required_documents": ["mil_id", "passport"]
}
```

Движок для каждой меры умеет ответить на 4 вопроса:
1. **Почему подходит** — список `matched` (какие критерии выполнены).
2. **Какие критерии выполнены** — те же `matched` с фактическими значениями.
3. **Чего не хватает / что не так** — `unmet` (нарушенные жёсткие) и `missing` (нет данных).
4. **Что сделать для подачи** — `next_step` + `required_documents` (из карточки меры).

Это же объяснение — вход для кабинета гражданина («что сделать сегодня») и для координатора.

### 6.1 Журнал оценки `RuleEvaluationLog` (согласовано PM)

Каждый прогон подбора пишется в неизменяемый журнал — для разбора спорных случаев
координатором/администратором и анализа качества правил. Гражданин журнал не видит.

| Поле | Тип | Назначение |
|------|-----|-----------|
| `id` | UUID | PK |
| `subject_type` | ENUM | citizen / organization |
| `project_id` | UUID (Н) | Контекст кейса |
| `user_id` / `organization_id` | UUID | Кого оценивали |
| `measure_id` | UUID | Оценённая мера |
| `measure_version` | INT | Версия меры на момент оценки |
| `score` | INT | Итоговый Score |
| `category` | ENUM | eligible / conditional / need_info / hidden |
| `unmet_rules` | JSONB | Какие жёсткие правила не прошли (причина скрытия) |
| `missing_fields` | JSONB | Каких данных не хватило |
| `engine_version` | STRING | Версия Rule Engine (воспроизводимость) |
| `evaluated_at` | TIMESTAMP | Время оценки |

Журнал наполняется на стадиях «Проверка критериев» и «Расчёт Score» для ВСЕХ мер, включая
скрытые (`hidden`). Это даёт координатору возможность вручную разобрать, почему мера не
показалась гражданину.

## 7. Масштабирование и эволюция

Архитектура заложена так, чтобы развиваться без переписывания модели данных:

| Этап | Что меняется | Что остаётся неизменным |
|------|--------------|--------------------------|
| MVP (сейчас) | JSON-интерпретатор `eligibility`, пороговый Score | Модель данных, формат `eligibility`, контракт Explainability |
| v2 — полноценный Rule Engine | Замена интерпретатора на Drools/собственный DSL, приоритеты правил, A/B, версии наборов правил | Формат хранения правил в БД, API подбора |
| v3 — ИИ-ассистент | Диалоговый сбор анкеты, подсказки формулировок | Детерминированное ядро подбора под капотом (AI не принимает юр. решение) |
| v4 — ML/предиктив | Скоринг-модель как **дополнительный** `soft`-сигнал; предсказание «что понадобится дальше» | Жёсткие критерии остаются rule-based и объяснимыми (регуляторное требование) |

**Ключевые точки расширения (спроектированы уже сейчас):**
- `eligibility.version` — версионирование формата правил.
- Разделение `rules` (жёсткие, детерминированные) и `soft`/ML-сигналы (ранжирование) — ML
  никогда не влияет на допуск, только на порядок. Это сохраняет объяснимость и юр. чистоту.
- Evaluation Context — плоский расширяемый словарь: новое поле подбора не ломает старые правила.
- Rule Engine отделён от Journey/Workflow — возвращает набор мер, дальше их подхватывает
  маршрут сопровождения.

## 9. Допущения MVP (что НЕ входит)

- Приоритизация «федеральное перекрывается региональным» реализована упрощённо (ранжирование
  §5), полноценный механизм конфликтов правил — v2.
- A/B-тестирование правил, версии **наборов** правил — v2.
- ML-скоринг и предиктивные рекомендации — v4 (сейчас только пороговая rule-based шкала).
- Диалоговый ИИ-сбор анкеты — v3 (сейчас статичная форма).
- Кэш-инвалидация при массовом обновлении мер описана концептуально, не детализирована.
